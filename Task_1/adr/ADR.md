# ADR-001: Модульный монолит

**Дата:** 2026-06-23

**Статус:** Accepted

## Контекст

AdScale  платформа Real-Time Bidding для управления рекламными аукционами. Текущая система представляет собой монолит на полиглотной архитектуре (C++, Go, Python, Node.js) с единой инстанцией PostgreSQL. Система обрабатывает 15 тысяч RPS в пике, но P95 latency составляет 110-120 миллисекунд (требование: ≤100 миллисекунд).

Ключевые архитектурные драйверы:
- **Производительность:** текущий P95 110-120ms превышает требование ≤100ms, нужно улучшение на 50%
- **Масштабируемость:** планируется рост от 15K RPS (текущее)  18K RPS (MVP, 3 месяца)  50K RPS (целевое, 1 год)
- **Доступность:** целевая SLA 99.5% (MVP)  99.9% (Target). Текущий монолит  SPOF (одна база данных)
- **Сроки:** MVP должна быть готова за 3 месяца, Target  за 1 год
- **Команда:** 35 разработчиков, нет выделенных SRE, ограниченный бюджет
- **Требование:** zero-downtime миграция. Нельзя отключать систему для больших изменений

Рассматривались три стратегии эволюции:

### Альтернатива 1: Big Bang Rewrite (полная переписка на микросервисы)
**Плюсы:** чистая архитектура, оптимальный дизайн с нуля.
**Минусы:** очень высокий риск (новая система может упасть в production), 6-12 месяцев разработки, невозможно запустить параллельно с текущей, потеря накопленного знания о production-поведении, все клиенты и DSP-партнеры переходят на новую систему одновременно (катастрофальный риск).

### Альтернатива 2: Остаться в монолите навсегда
**Плюсы:** минимальные изменения, быстрый выход в production MVP.
**Минусы:** масштабируемость упирается в единую БД (SPOF), невозможно масштабировать отдельные компоненты независимо, при 50K RPS система не потянет, команда остается с большим легаси-монолитом, сложно нанимать новых разработчиков (монолиты непопулярны).

### Альтернатива 3: Модульный монолит (постепенная миграция)
**Плюсы:** текущая система продолжает работать (zero-downtime), можно тестировать новые компоненты параллельно, поэтапно мигрировать трафик, откатываться если что-то сломается, минимизировать риск.
**Минусы:** требует дополнительной инфраструктуры (Redis, Kafka, read-replicas), нужна дисциплина в разделении старого и нового кода, переходный период может быть запутанным.

## Решение

**Выбран модульный монолит** для постепенной эволюции архитектуры AdScale от монолита к микросервисам за 12 месяцев.

### Фазы внедрения:

**Фаза 1: MVP (3 месяца)  Модульный монолит с инфраструктурой**
- Монолит остается основной системой (zero-downtime)
- Добавляется инфраструктура: Redis (кэш), Kafka (event streaming), PostgreSQL Master-Slave репликация
- Bidding Service (Auction Engine) выделяется как отдельный модуль внутри монолита
- Результат: P95 latency 110ms  50-60ms, 18K RPS достижима, SLA 99.5%

**Фаза 2: Масштабирование (6 месяцев)**
- PostgreSQL sharding (2 shards)
- Kafka расширяется до 3-broker HA cluster
- Постепенно мигрируется трафик на оптимизированные пути

**Фаза 3: Полные микросервисы (12 месяцев)**
- Bidding Service выделяется в отдельный микросервис
- Остальные компоненты постепенно становятся микросервисами
- Event-driven архитектура полностью активна
- Multi-region deployment готовность
- SLA 99.9% достигается

### Ключевые принципы:
- **Постепенность:** каждый месяц один или два небольших изменения, не революционные скачки
- **Дуплирование:** новые пути строятся рядом со старыми, трафик постепенно переходит
- **Откатываемость:** если новый компонент сломается, старый компонент всегда готов перехватить трафик
- **Мониторинг:** continuous A/B testing между старым и новым путями

## Последствия

### Положительные:
- **Zero-downtime:** система никогда не отключается, клиенты не видят перерывов
- **Минимизация риска:** риск распределяется по времени, каждая фаза может быть быстро отменена
- **Обучение:** команда учится на production данных, понимает real-world проблемы
- **Гибкость:** если обнаружатся новые требования, их легко интегрировать в текущую фазу
- **Ускоренный MVP:** за 3 месяца получаем работающую систему, соответствующую требованиям
- **Евангелизм:** успешный MVP сначала мотивирует команду и заинтересованные стороны на следующие фазы

### Отрицательные:
- **Временный перекрытие:** какое-то время нужно поддерживать оба пути (старый + новый)
- **Сложность для тестирования:** нужны интеграционные тесты для обеих систем одновременно
- **Техдолг:** новые слои инфраструктуры (Redis, Kafka) требуют операционной поддержки
- **Непредсказуемость:** сроки фаз 2 и 3 сложнее оценить в начале (зависит от learnings фазы 1)

### Риски:
- **Если MVP не соответствует SLA:** могут потребоваться более агрессивные изменения в фазе 1. Рискуем не уложиться в 3 месяца
- **Если появятся новые DSP-партнеры с нестандартными требованиями:** нужно гибко адаптировать архитектуру (но Strangler Fig это позволяет)
- **Если команда потеряет дисциплину:** техдолг накопится, качество упадет. Требуется code review и архитектурный надзор

# ADR-002: Bidding Service как первый приоритет выделения из монолита

**Дата:** 2026-06-23

**Статус:** Accepted

## Контекст

В рамках Strangler Fig Pattern нужно выбрать, какой компонент выделять первым. Кандидаты:
1. **Bidding Service** (Auction Engine)  ядро RTB платформы
2. **Analytics Service**  аналитика и отчеты
3. **Campaign Management**  управление кампаниями
4. **Financial Service**  обработка платежей

Критерии выбора:
- **Критичность:** компонент должен быть узким местом (bottleneck)
- **Независимость:** компонент должен работать с минимальными зависимостями
- **Изолируемость:** можно выделить четкие границы API
- **Impact:** выделение должно дать ощутимый прирост производительности

## Решение

**Выбран Bidding Service (Auction Engine) как первый приоритет.**

### Обоснование:

**Критичность:** Bidding Service  это горячий путь (hot path). Каждый bid request проходит через Auction Engine. Текущая производительность этого компонента напрямую определяет P95 latency системы (110-120ms vs требуемых ≤100ms). Оптимизация Bidding Service даст наибольший прирост производительности.

**Независимость:** Bidding Service имеет четкие зависимости:
- Читает данные о кампаниях (через Campaign Service)
- Читает бюджеты (через Financial Service)
- Пишет события показов (через Kafka)
Все эти зависимости можно обернуть в публичные интерфейсы (Facade).

**Изолируемость:** Bidding Service логически отделен:
- Входной API: OpenRTB от DSP
- Выходной API: JSON-response с winning ad
- Внутренние операции: выбор кандидатов, аукцион, рендер

**Impact:** Оптимизация Bidding Service (добавление Redis кэша, Kafka events вместо sync writes) дает:
- P95 latency: 110ms  50-60ms (достижение SLA за 3 месяца MVP)
- Write contention: убирается (Kafka вместо sync writes)
- Масштабируемость: появляется возможность горизонтального масштабирования

### Преимущества выбора именно Bidding Service:
- **Видимый результат:** внешние клиенты (DSP-партнеры) сразу видят улучшение latency
- **Меньше зависимостей:** чем другие компоненты (Analytics требует перестройки всей аналитики БД)
- **Возможность переиспользования:** паттерны Bidding Service (кэш, async events) будут полезны для других компонентов
- **Командная мотивация:** успешное выделение первого сервиса мотивирует на остальные

## Последствия

### Положительные:
- Достижение целевого P95 latency ≤100ms за 3 месяца (MVP)
- Снижение write contention в PostgreSQL (Kafka вместо sync writes)
- Появление чистых границ (API contract) для будущего выделения в микросервис
- Возможность параллельного разработки других компонентов (они не зависят от Bidding Service оптимизации)

### Отрицательные:
- Требует добавления Redis (дополнительная инфраструктура для поддержки)
- Требует Kafka (еще одна система для управления)
- Нужна дисциплина в разделении кода (Bidding Service должен быть модулем, не смешиваться с остальным)

### Риски:
- **Если выделение займет больше 3 месяцев:** нужно пересмотреть scope или сроки
- **Если Redis станет bottleneck:** потребуется Redis Cluster (дополнительная сложность)
- **Если Cache invalidation сломается:** система будет возвращать старые данные. Нужны хорошие тесты

# ADR-003: Tech Stack для MVP: Kafka + Redis + PostgreSQL Master-Slave

**Дата:** 2026-06-23

**Статус:** Accepted

## Контекст

Для MVP решения нужно выбрать технологии для:
1. **Кэширования**  снижение нагрузки на БД для Bidding Service
2. **Event streaming**  асинхронная обработка показов/кликов вместо sync writes
3. **Database scaling**  обеспечение High Availability и read-heavy операций

Рассматривались альтернативы:

### Кэширование
- **Redis:** fast in-memory, простой API, готовая поддержка TTL, есть опыт в команде 
- **Memcached:** еще проще, но нет встроенной поддержки complex data structures
- **Местный in-process cache:** нет shared state между replicas

### Event Streaming
- **Kafka:** distributed, persistent, high throughput, может работать с 18K RPS, ecosystem зрелый 
- **RabbitMQ:** проще, но для 18K RPS нужна более сложная конфигурация
- **Redis Streams:** заманчиво (одна система вместо двух), но для production reliability недостаточно зрелый
- **AWS SQS:** cloud-only, нет гибкости

### Database Scaling
- **PostgreSQL Master-Slave репликация:** native PostgreSQL feature, async binary log, простой failover 
- **PostgreSQL Logical Replication:** сложнее, не нужна для MVP
- **MySQL Galera:** не выбирали (команда знает PostgreSQL лучше)
- **Shared-nothing архитектура (шардирование):** слишком сложно для MVP, отложили на фазу 2

## Решение


### Компоненты:

**Redis (Кэширование):**
- **Версия:** Redis 6.0+
- **Конфигурация:** single instance для MVP (1 сервер), Master-Slave репликация для backup
- **Использование:** campaigns_cache (5min TTL), budgets_cache (1min TTL), bid_floors_cache (30min TTL)
- **Hit ratio target:** 95%+ (уменьшит нагрузку на БД в 20 раз)
- **SLA:** 99.5% (graceful degradation если Redis недоступна  просто читаем из БД медленнее)

**Kafka (Event Streaming):**
- **Версия:** Kafka 2.8+
- **Конфигурация:** single broker для MVP (потом расширим на 3 brokers для HA)
- **Topics:**
  - impressions.recorded (10 partitions, 1-day retention)
  - clicks.recorded (10 partitions, 1-day retention)
  - transactions.completed (5 partitions, 7-day retention)
  - campaigns.updated (3 partitions, 1-hour retention)
- **Партиционирование:** по campaign_id для гарантии порядка
- **Throughput:** 18K RPS = ~100K messages/sec (single broker может обработать)

**PostgreSQL Master-Slave:**
- **Версия:** PostgreSQL 12+
- **Master:** OLTP writes (campaigns, budgets, bids, ledger). Все клиенты пишут в Master
- **Slave 1:** Read-only для Analytics (heavy queries не блокируют production)
- **Slave 2:** Hot standby для failover (если Master падает, promote Slave 2)
- **Репликация:** асинхронная binary log (1-2ms lag OK)
- **Failover:** автоматический или manual (в зависимости от SLA требований)

## Последствия

### Положительные:
- **Производительность:** P95 latency 110ms  50-60ms (кэш + async events)
- **Write scalability:** write contention убирается (Kafka вместо sync writes к БД)
- **Read scalability:** Analytics читает из read-replica, не конкурирует с production
- **Reliability:** Master-Slave дает HA (нет SPOF для БД)
- **Ecosystem:** все три технологии (Redis, Kafka, PostgreSQL) mature, есть community support
- **Cost:** относительно бюджетно (single instances нужны для MVP, можем расширить потом)

### Отрицательные:
- **Operational complexity:** добавляется Redis и Kafka для поддержки
- **Data consistency:** Kafka events не гарантируют exactly-once (нужно deduplication на consumer side)
- **Cache invalidation:** нужны правила для TTL и event-based invalidation
- **Network latency:** Redis + Kafka добавляют network round-trip, но амортизируется throughput

### Риски:
- **Если Redis становится bottleneck:** потребуется Redis Cluster (фаза 2, дополнительная сложность)
- **Если Kafka throughput недостаточно:** потребуется масштабировать на 3-5 brokers (фаза 2)
- **Если PostgreSQL Master падает:** нужна пара минут на failover. RTO ~60sec (приемлемо для MVP)

### Миграция стратегия:
1. **Месяц 1:** развернуть Redis, Kafka, PostgreSQL Slave. Все еще работает со старыми путями
2. **Месяц 2:** постепенно мигрировать трафик на новые пути (Redis + Kafka)
3. **Месяц 3:** финальная оптимизация, performance testing, готовность к production
