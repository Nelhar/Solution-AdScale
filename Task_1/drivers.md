# Архитектурные Драйверы AdScale

##  Функциональные Требования

AdScale  это платформа для управления Real-Time Bidding (Bidding Services) рекламных аукционов. Основные возможности включают управление рекламными кампаниями, бюджетами и ставками, интеграцию с DSP-платформами через протокол OpenRTB, показ рекламы пользователям веб-сайтов и мобильных приложений, а также сбор аналитики по кликам, показам и доходам.

Ключевая интеграция с DSP-партнерами предполагает прием bid requests и отправку bid responses по OpenRTB протоколу. При интеграции новой крупной DSP-платформы ожидается увеличение трафика на 20 процентов в краткосрочной перспективе и двукратный рост в течение года.

##  Нефункциональные Требования

### Производительность

Текущая нагрузка составляет 15 тысяч запросов в секунду (RPS) в пиковые часы. В то же время требование по latency критично  P95 latency должна быть не более 100 миллисекунд, что сейчас не достигается (текущий P95 составляет 110-120 миллисекунд). После внедрения улучшений в MVP нужно поддерживать 18 тысяч RPS с P95 latency не более 100 миллисекунд. На целевом этапе система должна масштабироваться до 50 тысяч RPS с P99 latency не более 80 миллисекунд.

### Надежность

Система обрабатывает финансовые операции, поэтому потеря данных недопустима. Любой downtime критично влияет на репутацию компании и приносит финансовые убытки. Нарушение SLA соглашений может привести к штрафам и потере контрактов с DSP-партнерами.

### Масштабируемость

Система должна позволять горизонтальное добавление серверов для увеличения пропускной способности. В настоящий момент отсутствует запас производительности, поэтому требуется оптимизировать архитектуру перед ростом нагрузки. Растущий объем данных усложняет работу с базой данных, особенно для аналитических запросов.

## � Выявленные Проблемы

### CRITICAL: Auction Engine как узкое место

Auction Engine выполняет синхронные вызовы к PostgreSQL в самом критичном пути обработки bid request. Каждый вызов к базе данных занимает 5-10 миллисекунд. Поскольку требование по P95 latency составляет максимум 100 миллисекунд, а сейчас система показывает 110-120 миллисекунд, появляется необходимость срочной оптимизации. Каждые дополнительные 10 миллисекунд задержки означают потерю возможности выиграть аукцион.

Решение заключается в кэшировании данных о кампаниях в Redis, что может снизить latency с 110 миллисекунд до 50 миллисекунд.

### CRITICAL: PostgreSQL как единственная точка отказа

В текущей архитектуре используется одна инстанция PostgreSQL, которая содержит все данные системы. При падении этой инстанции вся система становится недоступной. Отсутствуют механизмы репликации, failover и высокой доступности. Это создает критический риск для бизнеса.

Требуется внедрить репликацию Master-Slave, при которой основная база данных выполняет запись, а реплика используется только для чтения. При необходимости реплика может быть повышена до роли primary сервера.

### HIGH: Statistics Service создает contention при записи

Statistics Service записывает информацию о каждом impression и click синхронно напрямую в базу данных. При 15 тысячах RPS это означает 15 тысяч INSERT операций в секунду, что создает блокировку ресурсов и конкуренцию за доступ к таблицам. Это напрямую влияет на Auction Engine, замедляя его работу.

Решение состоит в использовании Kafka для асинхронной буферизации событий. RTB Service будет отправлять события в Kafka (операция занимает ~1 миллисекунду), а затем отдельные consumer приложения будут обрабатывать эти события в своем темпе, без блокировки критичного пути.

### HIGH: Analytics блокирует основные операции

Analytics Service выполняет тяжелые SQL запросы (например, агрегацию по 50 миллионам строк) напрямую на production базе данных. Эти запросы конкурируют за CPU, память и I/O ресурсы с критичными операциями Auction Engine, вызывая замедление на несколько секунд.

Требуется отделить Analytics от основной базы данных, используя read-only реплику специально для аналитических запросов.

##  Бизнес-Ограничения

MVP версия должна быть готова за 3 месяца и должна поддерживать 18 тысяч RPS с P95 latency не более 100 миллисекунд. Целевая версия (полная микросервисная архитектура с 50 тысячами RPS) должна быть готова за 1 год.

Команда состоит из 35 человек без выделенных специалистов по надежности систем (SRE) и технической документации. Бюджет ограничен, поэтому решение должно быть максимально эффективным по затратам.

Важное требование  система должна оставаться работающей без перерывов. Невозможно отключить систему для больших изменений (zero-downtime deployment требуется). Миграция должна быть постепенной с возможностью отката.

##  Приоритизированные Проблемы

Самыми критичными проблемами являются Auction Engine bottleneck, PostgreSQL SPOF, и write contention в Statistics Service. Эти три проблемы напрямую влияют на latency и SLA.

Важными, но менее критичными, являются отсутствие кэша, блокировка Analytics, и отсутствие подготовки к sharding для будущего роста.

Среднего приоритета задачи включают разделение на микросервисы, внедрение event-driven архитектуры и улучшение мониторинга.

##  Критерии Успеха

Для MVP версии требуется достичь 18 тысяч RPS, P95 latency не более 100 миллисекунд, новая DSP должна интегрироваться без деградации performance, текущие клиенты должны работать стабильно, и система должна обеспечивать 99.5% SLA.

Для целевой версии система должна масштабироваться до 50 тысяч RPS с 99.9% SLA, разделяться на независимые микросервисы, обрабатывать события асинхронно, и быть готовой к развертыванию в нескольких географических регионах.

##  Стратегия Эволюции

Буде применен Strangler Fig Pattern для постепенной миграции. MVP версия будет представлять собой модульный монолит с добавлением инфраструктуры Redis для кэширования, Kafka для event streaming, и Master-Slave репликацией PostgreSQL.

На следующем этапе будет выполнено выделение RTB Service как отдельного микросервиса. На целевом этапе система будет полностью разделена на микросервисы с Event Sourcing и CQRS паттернами.
